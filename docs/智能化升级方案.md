# 灵感大王 · 智能化升级路径（2026-07-15）

> **背景**：用户反馈"灵感大王不够智能，除了接入 API，还有什么办法？"
> **本调研目标**：盘点**非云端 LLM**或**混合架构**的智能化升级方案，评估技术边界，给出渐进式实施路径。
> **核心结论**：**LLM 不是唯一的"智能"**——本地小模型 + 规则引擎 + RAG + 用户行为学习 + 多 Agent 编排，可以构建**"零成本/低成本 + 离线可用 + 隐私安全"**的复合智能体系。

---

## 1. 为什么需要"无 LLM"路径

云端 LLM 的三大痛点：
- **成本**：企业 LLM API 2025 年规模 $8.4B，单个重度用户年成本 $1k-$36k
- **隐私**：44% 组织视数据隐私为 LLM 采纳首要障碍（2025）
- **稳定**：2025 年 OpenAI API 中断期间，基于本地 SLM 的应用正常运行
- **延迟**：云端 token-by-token 网络往返，本地首字延迟 <100ms

但本地方案的短板同样存在：
- 中文质量参差（1B 较差，3B 可用，7B+ 才有竞争力）
- 硬件门槛（4-bit 量化下 7B 需 4GB 显存/8GB RAM）
- 模型分发与首次下载体验
- 复杂推理不及云端

**最佳策略不是二选一，而是"分层编排"：高频轻量走本地/规则，复杂深度走云端 LLM**。

## 2. 六大智能化方案对比

### 方案 A：本地小模型（On-device SLM）★ 推荐重点

| 维度 | 说明 |
|------|------|
| **代表模型** | Llama 3.2 1B/3B、Phi-3 Mini (3.8B)、Qwen2.5 1.5B、Gemma 2 2B、DeepSeek-coder 1.3B |
| **技术栈** | llama.cpp / Ollama / LocalAI / Cherry Studio / @xenova/transformers（浏览器端） |
| **集成方式** | Tauri sidecar 启动 `llama-server`（OpenAI 兼容 API），复用现有 OpenAIAdapter |
| **模型体积** | 1B ≈ 1.2GB / 3B ≈ 2GB（Q4_K_M 量化） |
| **硬件门槛** | 8GB RAM（1-3B）/ 16GB RAM（7B）/ 24GB VRAM（13B+） |
| **中文质量** | Qwen2.5 1.5B > Phi-3 Mini > Llama 3.2 1B（中文） |
| **首字延迟** | <100ms（本地）/ 300-800ms（云端） |
| **优势** | 隐私（数据不出机器）、离线可用、零 API 成本、低延迟 |
| **劣势** | 硬件门槛、中文质量参差、模型分发与首次下载 |
| **适用场景** | "直接生成"模式、"快速模式"（7 题）、追问触发判断 |
| **不适场景** | 31 题全流程深度诊断、复杂需求拆解 |

### 方案 B：规则引擎 + 模板检索（RAG-Lite）★ 立即可做

| 维度 | 说明 |
|------|------|
| **核心思路** | 扩展现有 seedRouter + 评价词典 + TagCollector，引入模板库与历史答案检索 |
| **已有基础** | seedRouter 否定词识别、TagCollector 依赖注入、contextStore 偏好累积 |
| **新增能力** | (1) 每种场景预置 3-5 套高质量 prompt 模板；(2) 历史答案检索（"上次你怎么处理圆角？"）；(3) 聚类挖掘（从用户答案库找相似项目自动推荐） |
| **优势** | 零依赖、零延迟（<1ms）、可解释、零成本、纯本地 |
| **劣势** | 覆盖场景有限（需持续维护规则库）、无法处理开放域问题 |
| **适用场景** | 垂直场景（前端 UI/后端 API）已经够用，是当前规则模式的扩展 |
| **工作量** | 0.5-1d（数据驱动，非工程问题） |

**实操样例**：

```typescript
// src/engine/TemplateRetriever.ts（新增）
export async function retrieveSimilarTemplate(
  seed: string,
  history: HistoryEntry[],
  templatePool: PromptTemplate[]
): Promise<PromptTemplate | null> {
  // 1. 提取 seed 的关键 tag
  const tags = await extractTags(seed);
  // 2. 计算与历史答案的 tag 相似度
  const scored = templatePool.map(t => ({
    template: t,
    score: jaccardSimilarity(t.tags, tags) * 0.5 +
           historyBoost(t, history) * 0.3 +
           t.popularity * 0.2
  }));
  return scored.sort((a, b) => b.score - a.score)[0]?.template ?? null;
}
```

### 方案 C：知识图谱 + 领域本体

| 维度 | 说明 |
|------|------|
| **核心思路** | 把设计规范、UX 原则、组件库特性构建成本体图，支持推理 |
| **代表项目** | ConceptNet、WordNet、UMLS、CYC、谷歌 Knowledge Graph |
| **新增能力** | "圆角" → 关联"亲和力""现代感""Material Design"；"去圆角" → 关联"严肃""工业""Windows 11" |
| **优势** | 可解释、推理强、零 LLM 依赖 |
| **劣势** | 构建成本极高、需要领域专家、覆盖率瓶颈 |
| **适用场景** | B 端、企业知识管理、垂直行业（医疗/法律） |
| **不推荐** | C 端工具，构建 ROI 极低 |

**技术边界**：知识图谱构建成本通常 100-500 人日（中型本体），灵感大王场景下投入产出比不划算。

### 方案 D：用户行为学习（Personalization）★ 立即可做

| 维度 | 说明 |
|------|------|
| **核心思路** | 从用户历史答案中挖掘偏好，提供个性化推荐 |
| **已有基础** | contextStore.MAX_PREFERENCES=30、analyticsStore.MAX_RECORDS=500 |
| **新增能力** | (1) 频率分析："用户 A 80% 选'简洁'"；(2) 时序趋势："最近 5 次选了'鲜艳'，可能转变"；(3) 跨会话学习：昨天答的偏好，今天直接预填 |
| **技术栈** | 纯本地计算（SQLite + 简单统计/聚类） |
| **优势** | 隐私好、零成本、可渐进优化 |
| **劣势** | 冷启动难（需要 ≥5 个会话才有统计意义）、需要数据保留策略 |
| **适用场景** | 复访用户、长期使用、个性化预填 |
| **工作量** | 1-2d（前端 + 算法） |

**实操样例**：

```typescript
// src/engine/PreferenceLearner.ts（新增）
export interface UserProfile {
  topChoices: Map<string, { optionId: string; count: number; ratio: number }>;
  trends: Array<{ tag: string; direction: 'up' | 'down' | 'stable' }>;
  patterns: Array<{ rule: string; confidence: number }>;
}

export function buildProfile(prefs: PreferenceEntry[]): UserProfile {
  // 1. 频率统计：每题最常选
  const topChoices = groupBy(prefs, p => p.questionId)
    .map(([qid, entries]) => ({
      questionId: qid,
      topOption: mostFrequent(entries.map(e => e.optionId)),
      count: entries.length
    }));
  // 2. 时序趋势：最近 vs 历史的对比
  const trends = detectTrends(prefs);
  return { topChoices, trends, patterns: [] };
}
```

### 方案 E：多 Agent 协作（本地 + 远程混合）★ 长期演进

| 维度 | 说明 |
|------|------|
| **核心思路** | 本地 SLM 处理轻量任务（意图分类、追问触发），远程 LLM 处理复杂任务（深度拆解） |
| **参考架构** | Agent 1（本地 SLM）→ 意图分类 + 追问判断；Agent 2（云端 LLM）→ 深度拆解；Agent 3（本地规则）→ 模板拼装 + 约束校验 |
| **优势** | 成本与质量平衡、灵活降级链路 |
| **劣势** | 架构复杂、调试困难、需要清晰的边界协议 |
| **适用场景** | 高级用户、Pro 版、高频用户（用量起来后成本敏感） |
| **工作量** | 5-10d（架构设计为主） |

**调度策略草图**：

```
用户输入
  ↓
[本地 SLM 1B] 意图分类（<50ms）
  ├─ 明确 → [本地规则] 直接生成（0ms）
  ├─ 中等 → [本地 SLM 3B] 快速模式 7 题（2-3s）
  └─ 复杂 → [云端 LLM] 详细模式 31 题（5-10s）
                ↓ 失败
              [本地 SLM 3B] 降级
                ↓ 失败
              [本地规则] 兜底
```

### 方案 F：检索增强生成（RAG）+ 本地向量库 ★ 中期推荐

| 维度 | 说明 |
|------|------|
| **核心思路** | 把用户历史答案、设计规范、组件文档向量化，提问时检索 Top-K 增强上下文 |
| **技术栈** | 本地嵌入：@xenova/transformers（浏览器端）/ sqlite-vss；模型：all-MiniLM-L6-v2 (90MB) / BGE-small-zh-v1.5 (93MB) |
| **新增能力** | 跨会话上下文复用、避免重复提问、引用规范文档 |
| **优势** | 增强 LLM 上下文、显著提升质量、模型小（<100MB） |
| **劣势** | 首次加载 5-10s、需 200-500MB 额外存储、向量化计算开销 |
| **适用场景** | 项目级、跨会话、领域知识查询 |
| **工作量** | 3-5d（嵌入模型集成 + 索引构建 + 检索集成） |

**实操样例**：

```typescript
// src/lib/vectorStore.ts（新增）
import { pipeline } from '@xenova/transformers';

const embedder = await pipeline('feature-extraction', 'Xenova/bge-small-zh-v1.5');

export async function embed(text: string): Promise<Float32Array> {
  const result = await embedder(text, { pooling: 'mean', normalize: true });
  return new Float32Array(result.data);
}

export async function retrieveContext(
  query: string,
  history: HistoryEntry[],
  topK: number = 5
): Promise<HistoryEntry[]> {
  const queryVec = await embed(query);
  // 余弦相似度排序
  const scored = history.map(h => ({
    entry: h,
    score: cosineSimilarity(queryVec, h.embedding)
  }));
  return scored.sort((a, b) => b.score - a.score).slice(0, topK).map(s => s.entry);
}
```

## 4. 推荐实施路径（渐进式）

### Phase A：立即（1-2 周）—— 零成本增强

- **方案 B**（规则+模板）：扩展 seedRouter，添加 5-10 套场景模板库
- **方案 D**（行为学习）：buildProfile() 实现频率统计，跨会话预填
- **预期效果**："无 LLM"模式质量提升 30-50%
- **投资**：0 元 / 1.5-3d

### Phase B：短期（1-2 月）—— 本地 SLM 集成

- **方案 A**：llama.cpp + GGUF Q4_K_M 模型
- 推荐模型起点：**Qwen2.5 1.5B**（中文友好）/ **Phi-3 Mini 3.8B**（英文更强）
- Tauri sidecar 启动 `llama-server`（OpenAI 兼容），复用 OpenAIAdapter
- 应用场景："直接生成" + "快速模式"走本地；"详细模式"仍走云端
- **预期效果**：80% 场景零 API 成本、离线可用、首字延迟 <100ms
- **投资**：0 元 / 3-5d + 1-2GB 模型分发

### Phase C：中期（2-3 月）—— RAG 增强

- **方案 F**：本地嵌入 + sqlite-vss
- 嵌入模型：BGE-small-zh-v1.5（93MB，CPU 可跑）
- 应用场景：历史答案向量库、问题模板库、跨会话上下文增强
- **预期效果**：复用历史答案，避免重复提问；提示词质量再提升 20%
- **投资**：0 元 / 3-5d + 100-200MB 嵌入模型

### Phase D：长期（3+ 月）—— 多 Agent 编排

- **方案 E**：轻量任务走本地，复杂任务走云端
- 调度策略：意图分类 → 本地 SLM → 决策路由 → 云端 LLM / 本地规则兜底
- **预期效果**：成本与质量的最优平衡；高级用户可解锁 Pro 功能
- **投资**：中等 / 5-10d

## 5. 技术边界与决策矩阵

### 何时用 LLM API

- ✅ 用户已配置云端 API Key
- ✅ 任务复杂（31 题全流程、需求深度拆解）
- ✅ 网络稳定
- ✅ 用户对延迟不敏感（5-10s 可接受）

### 何时用本地 SLM

- ✅ 用户未配置 API Key
- ✅ 用户对隐私敏感（项目代码/商业机密）
- ✅ 离线场景（出差、飞机、保密环境）
- ✅ 高频调用（每天 ≥10 次，节省 API 成本）
- ✅ 硬件满足（8GB+ RAM）

### 何时用规则引擎

- ✅ 启动阶段（冷启动，零配置）
- ✅ 高频/低复杂度任务（意图分类、模板拼装）
- ✅ 需要可解释性
- ✅ LLM 失败降级

### 何时用 RAG

- ✅ 跨会话上下文复用
- ✅ 领域知识查询（设计规范、组件库）
- ✅ 用户答案库 ≥20 条

---

## 6. 关键决策建议

1. **不要追求"最强模型"，要追求"最合适模型"**——按场景分层
2. **优先投资零成本方案**（B + D），立刻见效
3. **本地 SLM 是"差异化"关键**——隐私 + 离线 + 零成本，是竞品没有的护城河
4. **RAG 优于调更大模型**——100MB 嵌入模型 + 检索，比加 1B 参数更有效
5. **多 Agent 是演进方向，不是起点**——架构复杂，先把单一 Agent 跑通
6. **保留"降级链路"**——云端失败 → 本地 SLM → 规则兜底，确保用户永远不卡

## 8. 一句话总结

**"灵感大王更智能"≠"调用更大模型"。** 真正的智能化是**分层编排**：规则 + 行为学习（零成本）→ 本地小模型（隐私+离线）→ RAG（上下文增强）→ 多 Agent（高级编排），最终实现"零配置可用、按需升级、永不卡死"的产品体验。
