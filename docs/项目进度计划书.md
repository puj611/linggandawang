#### 里程碑 M8.6：问题库选项用户化改写（06/24）✅ 已完成

**背景**：用户反馈输入"帮我修改前端风格"时问题跑偏；同时小白用户面对技术选项（Tailwind class / 内边距 16px / WCAG AA / H1-H4）只能乱选。通过行业调研（Typeform 对话式表单原则 + CAT 自适应测试 + RICH 设计范式），确定"用户视角语言 + 参照物"是最佳实践。

**改写原则**：
- `label` 全部改为用户能看懂的口语化描述，加参照物（"像 Linear/Notion/Apple 那样"）
- `tags` 保留技术细节给代码用（如像素值、WCAG 等级、Tailwind），不影响 PromptGenerator 逻辑
- 题干本身也白话化（如"圆角偏好哪种？"→"边角喜欢圆润还是方正？"）
- `placeholder` 改为"或者用你自己的话说..."替代生硬的"或输入..."

#### 里程碑 M9.1：提问快速模式（06/25）✅ 已完成

**背景**：用户反馈"某些场景不需要那么多轮询问，一直点跳过太麻烦。某些场景只是需要强化提示词"。完整 31 题流程对轻量场景过重，需要提供更灵活的提问深度选项。

**设计方案**：三档模式选择器

| 模式 | 行为 | 适用场景 |
|------|------|----------|
| 直接生成 | 不问任何问题，仅基于用户输入+意图关键词提取直接生成提示词 | 只需要强化提示词、简单需求、目标明确 |
| 快速提问 | 仅问 7 道核心题（p-001 需求类型 / n-001 乱的类型 / n-007 风格方向 / s-001 间距 / s-002 圆角 / s-005 色调 / e-001 影响范围） | 有明确方向但需要确认关键规格，快速出结果 |
| 详细诊断 | 完整 31 题流程（原有默认行为） | 不确定方向、需要深度诊断 |

**核心题选择依据**：覆盖五大阶段（perceive→name→spec→execute）中最影响提示词质量的关键决策点，跳过 verify 阶段（快速模式不纠结验证策略，auto 兜底），共约 7 题。

|------|------|--------|------|------|
| T9.1.1 | types.ts 新增 FlowMode 类型 + Question.quick_mode 字段 | P0 | 0.1d | ✅ 完成 |
| T9.1.2 | bank.yaml 标记 7 道核心题 quick_mode: true | P0 | 0.2d | ✅ 完成 |

**技术要点**：
- 快速模式下 jump 规则如果指向非核心题，自动跳过不跟随，回退到正常选题逻辑
- verify 阶段无核心题，快速模式自动跳过该阶段
- 直接生成模式仍执行种子关键词意图提取（extractFromSeed），保证提示词有基本方向性
- 模式选择器采用分段控件样式（segmented control），默认"详细诊断"，用户可切换
- 提交按钮文案随模式变化："直接生成 →" vs "开始提问 →"
