## 一、市场研究总结

### 1.1 行业趋势分析

基于 2026 年 AI 编程工具市场研究，关键趋势如下：

| 趋势 | 数据/描述 | 对灵感大王的启示 |
|------|-----------|-----------------|
| **Vibe Coding 已成主流** | 85% 开发者日常使用 AI 工具，60% 新代码由 AI 生成 | 产品定位准确，需深化"意图传达管线"差异化 |
| **Agent 模式普及率飙升** | 2026 年底预计达 70%+ | 需增强智能追问和自主决策能力 |
| **桌面 AI 助手兴起** | Google Gemini、GhostDesk、RED AI 纷纷推出桌面浮窗产品 | 浮窗形态已成为行业标配，需强化差异化功能 |
| **上下文工程成为核心竞争力** | 优秀的上下文管理使 AI 输出质量提升 80% | 深化项目扫描和上下文理解能力 |
| **PromptPerfect 即将关闭** | 2026 年 9 月 1 日关闭，市场出现空白 | 机会窗口：抢占提示词优化工具市场份额 |
| **隐私保护成为刚需** | 用户对本地运行、数据不上传的需求持续增长 | 强化本地优先架构，不收集用户数据 |

### 1.2 竞品分析（2026 年现状）

| 竞品 | 定位 | 核心优势 | 劣势 | 灵感大王差异化机会 |
|------|------|----------|------|-------------------|
| **Cursor** | AI 代码编辑器 | 深度代码理解、Composer 功能 | 重 IDE，需切换窗口 | 浮窗形态，无需切换 |
| **Claude Code** | 终端 Agent | 全仓库理解、自主执行 | CLI 交互，学习曲线陡 | 可视化界面，小白友好 |

| **RED AI** | 轻量浮窗助手 | 零摩擦、上下文感知 | 功能简单，无编码深度 | 专业提示词生成 + 项目上下文 |

## 六、技术规范

### 6.1 架构约束

| 约束 | 说明 |
|------|------|
| **分层架构** | UI → Hooks → Engine → Lib → Rust，禁止反向依赖 |
| **纯函数引擎** | engine 层必须是无状态纯函数，可独立测试 |
| **最小权限原则** | Tauri capabilities 仅授予必要权限 |
| **类型安全** | TypeScript strict 模式，禁止 `any`，使用 `unknown` + 类型守卫 |
| **异步安全** | async 操作需 mountedRef 保护，禁止组件卸载后 setState |
| **Zustand 订阅优化** | 使用逐个 selector 或 `useShallow`，禁止整 store 订阅 |

### 6.2 性能规范

| 指标 | 目标值 | 测量方式 |
|------|--------|----------|
| 启动时间 | < 2 秒 | 从进程启动到悬浮球显示 |
| 热键响应 | < 200ms | 按键到窗口展开 |
| 项目扫描 | < 10 秒（1000 文件） | 扫描进度条时间 |
| 内存占用 | ≤ 200MB | Windows 任务管理器 |
| 包体大小 | ≤ 20MB | 安装包文件大小 |

### 6.3 安全规范

| 规范 | 说明 |
|------|------|
| API Key 存储 | 使用 keyring（Windows Credential Manager） |
| 敏感文件过滤 | 跳过 `.env`、`*.pem`、`*.key` 等敏感文件 |
| CSP 配置 | `connect-src: https:`，防止 SSRF |
| 输入验证 | 所有外部输入必须验证，防止注入攻击 |
| 数据脱敏 | 错误信息中隐藏 API Key 和敏感路径 |

## 八、质量保证措施

### 8.1 测试策略

| 测试类型 | 覆盖范围 | 执行频率 |
|----------|----------|----------|
| **单元测试** | engine 层纯函数、lib 层工具函数 | 每次代码提交 |
| **集成测试** | 核心流程（输入→提问→生成→复制） | 每日构建 |
| **性能测试** | 启动时间、内存占用、扫描速度 | 里程碑完成后 |
| **安全测试** | 路径遍历、符号链接、API Key 泄露 | 里程碑完成后 |
| **手动测试** | UI 交互、边界条件、错误处理 | 每周 |

### 8.2 质量门禁

| 门禁 | 要求 | 检查方式 |
|------|------|----------|
| TypeScript 类型检查 | `tsc --noEmit` 0 错误 | CI 自动检查 |
| 单元测试 | `vitest run` 100% 通过 | CI 自动检查 |
| Rust 编译 | `cargo check` 通过 | CI 自动检查 |
| 前端构建 | `vite build` 成功 | CI 自动检查 |
| 代码审查 | 核心变更需 1 人审查 | PR 流程 |

### 8.3 验证脚本

使用现有 `scripts/verify.ps1` 进行一键验证：

```powershell
# 执行顺序
1. npx tsc --noEmit          # TypeScript 类型检查
2. npx vitest run            # 单元测试
3. cd src-tauri && cargo check  # Rust 编译检查
4. npx vite build            # 前端构建
```
