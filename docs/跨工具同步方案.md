## 0. 文档说明

本文档是「灵感大王」项目在多 vibe coding 工具（Trae IDE / Claude Code / WorkBuddy）间切换开发时的**协同规范**，是 `docs/架构设计.md` 第 9 章的详细展开版。供所有团队成员（PM、架构师、工程师、QA）在切换工具时直接参考，避免上下文丢失、进度错位、产物冲突。

**结论先行（TL;DR）**：
- **单一事实源**：`docs/` 目录是项目事实的唯一来源，所有工具的配置目录只放工具特定配置，不存放业务事实。
- **运行时上下文**：`~/.linggandawang/context.json` 承载跨工具共享的运行时状态（项目、痛点、偏好、进度），所有工具读写同一文件。
- **锚点清单**：每次切换工具必读 5 个文件，确保上下文对齐。
- **进度字段**：`context.json.dev_progress` 是跨工具进度同步的核心字段，离开工具前必须更新。
- **冲突仲裁**：docs 文档以 git 为准；context.json 以 mtime 为准；代码冲突走标准 git 合并。

---

## 1. 目标

1. **上下文不丢**：从 Trae 切到 Claude Code，能立即知道上次在做什么、卡在哪里、下一步是什么。
2. **进度可对齐**：每个工具都能读到当前里程碑、当前任务、阻塞原因，避免重复劳动。
3. **产物不冲突**：各工具的配置目录职责清晰，不互相覆盖业务事实。
4. **低摩擦**：切换工具时不需要手动同步，读取锚点清单即可续接。

## 2. 各工具配置目录映射

| 工具 | 配置目录 | 作用 | 是否提交 git | 谁维护 |
|---|---|---|---|---|
| 灵感大王 | `~/.linggandawang/` | 运行时上下文 + 用户配置 + 问题库覆盖 | 否（本地） | 应用自动 + 用户手动 |
| Trae IDE | `.trae/rules/` | Trae 项目规则、prompt 模板 | 是 | 架构师 |
| Claude Code | `.claude/agents/` `.claude/commands/` | Claude Code 自定义 agents 与 commands | 是 | 架构师 |
| WorkBuddy | `.workbuddy/`（如启用） | WorkBuddy 配置 | 是 | 架构师 |
| 单一事实源 | `docs/` | PRD、架构设计、跨工具同步方案 | 是 | PM + 架构师 |
| 代码 | `src/` `src-tauri/` | 源代码 | 是 | 工程师 |
| 交付物 | `deliverables/` | 自测报告、Demo 视频、脚本 | 是 | QA + 工程师 |

### 2.1 目录职责边界

- **`docs/`**：业务事实的**唯一事实源**。PRD、架构设计、跨工具同步方案只允许在 PM/架构师授权下修改，其他工具不得自动写入。
- **`.trae/` `.claude/` `.workbuddy/`**：**只放工具特定配置**（规则、prompt 模板、agent 定义），不存放业务事实。这些目录的内容应能从 `docs/` 派生或引用。
- **`~/.linggandawang/`**：**运行时数据**，跨工具共享。应用本身读写，工具切换时由人工或脚本触发读取。
- **`src/` `src-tauri/`**：代码资产，走 git 标准流程。

### 2.2 为什么不用 .editorconfig / .gitattributes 等通用配置？

通用配置（editorconfig、gitattributes、prettierrc）放在项目根目录，所有工具共享，不归入任何工具的配置目录。这些是**工程约定**，不是业务事实。

## 3. 上下文锚点清单（每次切换工具必读）

切换到任何工具后，**第一步**是读取以下 5 个文件，确保上下文对齐：

| 序号 | 文件路径 | 读取方式 | 用途 | 优先级 |
|---|---|---|---|---|
| 1 | `~/.linggandawang/context.json` | 应用 API / 直接 fs 读取 | 运行时上下文（项目、痛点、偏好、进度） | P0 |
| 2 | `docs/PRD.md` | 直接读取 | 产品需求，理解做什么 | P0 |
| 3 | `docs/架构设计.md` | 直接读取 | 架构与任务列表，理解怎么做 | P0 |
| 4 | `docs/跨工具同步方案.md` | 直接读取 | 同步规则，理解怎么协作 | P1 |
| 5 | `README.md` | 直接读取 | 项目简介，快速回忆 | P1 |

### 3.1 读取顺序

1. **先读 context.json**：定位 `dev_progress.current_task_id`，知道上次在做什么。
2. **再读架构设计.md 第 6 章**：定位该任务的完整描述（输入、输出、依赖、验收标准）。
3. **按需读 PRD.md**：如果任务涉及需求细节，查对应 FR。
4. **读跨工具同步方案.md**：本文件，确认协作规则。

### 3.2 锚点清单的维护

锚点清单本身是固定的（5 个文件），不需要动态维护。如果未来增加新工具，新工具的配置目录自动加入"工具配置目录映射"表，但不影响锚点清单。

## 4. 进度同步状态字段

`context.json.dev_progress` 是跨工具进度同步的核心字段，定义如下：

### 4.1 字段定义

```json
"dev_progress": {
  "current_milestone": {
    "type": "string",
    "description": "当前里程碑 ID，对应架构设计第 6 章 M1-M7",
    "example": "M3"
  },
  "current_task_id": {
    "type": "string",
    "description": "当前任务 ID，对应架构设计第 6 章 T01-T30",
    "example": "T07"
  },
  "current_task_status": {
    "type": "string",
    "enum": ["todo", "in_progress", "blocked", "done"],
    "description": "当前任务状态",
    "example": "in_progress"
  },
  "blocked_reason": {
    "type": "string",
    "description": "如果状态是 blocked，填写阻塞原因；否则为空字符串",
    "example": "Alt+Space 热键被其他应用抢占，待排查"
  },
  "last_tool": {
    "type": "string",
    "enum": ["trae", "claude-code", "workbuddy", "manual"],
    "description": "上次使用的工具，便于追溯",
    "example": "trae"
  },
  "notes": {
    "type": "string",
    "description": "给下一个工具的交接说明，包括已尝试方案、待验证假设、注意事项",
    "example": "已尝试 tauri-plugin-global-shortcut 2.0，注册成功但 Win11 偶发不响应，怀疑是系统焦点问题，下一步用 Win32 API RegisterHotKey 兜底"
  }
}
```

### 4.2 维护时机

| 时机 | 操作 | 责任人 |
|---|---|---|
| 开始一个新任务 | 更新 `current_task_id`、`current_task_status='in_progress'`、`last_tool` | 当前工具操作者 |
| 任务进行中遇到阻塞 | `current_task_status='blocked'`、填写 `blocked_reason`、`notes` | 当前工具操作者 |
| 任务完成 | `current_task_status='done'`，清空 `blocked_reason`，更新 `notes` 说明产出 | 当前工具操作者 |
| 切换工具前 | 必须更新 `last_tool` 与 `notes`，确保下一个工具能续接 | 当前工具操作者 |
| 进入新里程碑 | 更新 `current_milestone` | 架构师 |

### 4.3 读取与续接流程

切换到新工具后，执行以下流程：

```
1. 读取 ~/.linggandawang/context.json
2. 定位 dev_progress.current_task_id
3. 在 docs/架构设计.md 第 6 章查找该任务的完整描述
4. 输出续接摘要：
   - 当前里程碑：{current_milestone}
   - 当前任务：{current_task_id} {任务名}
   - 状态：{current_task_status}
   - 上次工具：{last_tool}
   - 阻塞：{blocked_reason || "无"}
   - 交接说明：{notes}
5. 询问用户：是否继续该任务？
6. 用户确认后，开始执行
7. 离开工具前，更新 dev_progress 字段
```

### 4.4 字段示例

**示例 1：正常进行中**

```json

  "current_milestone": "M3",
  "current_task_id": "T07",
```

## 5. 冲突处理规则

### 5.1 文档冲突（docs/）

- **单一事实源原则**：`docs/PRD.md` 由 PM 许清楚维护，`docs/架构设计.md` 与 `docs/跨工具同步方案.md` 由架构师高见远维护。
- **修改流程**：在任意工具中修改 → 提交 git → 其他工具拉取最新。
- **冲突仲裁**：以 git 历史为准，冲突时由文档负责人裁决。
- **禁止**：工具自动修改 docs/ 下文档。

### 5.2 context.json 冲突

- **mtime 优先**：以文件 mtime 为准，最新覆盖最旧。
- **并发保护**：灵感大王应用启动时检测 mtime，若发现外部修改（其他工具手动编辑），以磁盘为准，不覆盖。
- **归档机制**：超过 24h 未更新的 context.json 自动归档到 `~/.linggandawang/archive/{timestamp}.json`，避免陈旧数据误导。

### 5.3 代码冲突（src/ src-tauri/）

- 走 git 标准合并流程，不在此文档约定。
- 建议：每个任务一个分支，完成后合并 main，避免长期分叉。

### 5.4 工具配置冲突（.trae/ .claude/ .workbuddy/）

- 各工具配置独立，互不干扰。
- 如果多个工具的配置有重叠（如都定义了项目规则），以 `docs/` 为准，工具配置引用 docs。

```markdown
灵感大王是一款桌面悬浮窗工具，通过反向提问把 vibe coding 新人的模糊灵感挤成精准提示词。

- 后端：Rust（最小化，只承担窗口/热键/文件 I/O/截图图像处理）

- 前端组件文件名 PascalCase（如 ExpandedCard.tsx）

- 不修改 docs/ 下任何文档（由 PM/架构师维护）
- 不引入未在架构设计第 7 章列出的依赖

- 界面字体 PingFang SC / Inter；代码字体 JetBrains Mono
```

## 7. 切换工具操作 SOP

### 7.1 离开当前工具前

1. **完成当前原子任务**或记录阻塞点到 `dev_progress.notes`
2. **更新 `~/.linggandawang/context.json`**：
   - `dev_progress.current_task_status`（todo/in_progress/blocked/done）
   - `dev_progress.last_tool`（当前工具名）
   - `dev_progress.notes`（交接说明：已做了什么、下一步做什么、注意事项）
   - `timestamps.updated_at`（应用自动更新）
3. **提交代码到 git**（如有改动）
4. **关闭工具**

### 7.2 进入新工具后

1. **执行 `/inspire` 命令**（或手动读取锚点清单）
2. **阅读续接摘要**，确认当前任务与状态
3. **如状态为 blocked**：先评估阻塞是否已解决，若解决则改为 in_progress
4. **继续执行任务**
5. **完成后更新 dev_progress**

### 7.3 工具切换矩阵

| 从 → 到 | 操作 |
|---|---|
| Trae → Claude Code | Trae 中保存文件 + 更新 dev_progress；Claude Code 执行 /inspire |
| Claude Code → Trae | Claude Code 更新 dev_progress + 提交 git；Trae 打开后读取 .trae/rules/project_rules.md 自动加载 |
| 任意工具 → WorkBuddy | 同上，WorkBuddy 读取 .workbuddy/ 配置（如有）+ context.json |
| 任意工具 → 灵感大王应用 | 灵感大王启动时自动读取 context.json，显示续接上次会话提示 |
